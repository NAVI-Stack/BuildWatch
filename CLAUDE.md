# BuildWatch

Universal file watcher & build daemon. Part of the NAVI Ecosystem but fully standalone.

## Build & Test

```bash
cargo build            # Debug build
cargo build --release  # Release build
cargo test             # Run all tests
cargo clippy           # Lint
cargo fmt              # Format
```

## Architecture

Read `docs/architecture.md` before making changes — it is the canonical specification.

## Project Structure

```
src/
├── main.rs        # CLI entry (clap) — `buildwatch` binary
├── bin/wr.rs      # Freshness gate — `wr` binary
├── lib.rs         # Shared library root (error types, project_hash, state_dir)
├── config.rs      # Config parsing (buildwatch.config.json)
├── daemon.rs      # Background daemon lifecycle & orchestration loop
├── watcher.rs     # Watchman integration (subscriptions, state assertions)
├── queue.rs       # Build queue with priority scoring & coalescing
├── builder.rs     # Build execution engine (timeout, log rotation, output capture)
├── detector.rs    # Project type auto-detection from filesystem markers
├── state.rs       # Shared state (atomic JSON files, lock, global registry)
├── notifier.rs    # System notifications (notify-rust)
└── output.rs      # Terminal formatting & agent detection
```

## Key Conventions

- **Config format:** JSON (`buildwatch.config.json`), generated by `buildwatch init`
- **State directory:** `/tmp/buildwatch/<project-hash>/` (Unix) or `%TEMP%\buildwatch\<hash>\` (Windows)
- **Atomic writes:** All state file writes use tempfile + rename. Never write directly.
- **One daemon per project root.** Enforced by advisory lock file.
- **`wr` binary is read-only.** It reads state files — never writes, never starts a daemon.
- **Platform guards:** Use `#[cfg(unix)]` / `#[cfg(windows)]` for platform-specific code, not runtime checks.

## Naming

- The project was renamed from "Wraith" to "BuildWatch". Do not use the old name.
- Binary names: `buildwatch` (daemon/CLI) and `wr` (freshness gate runner).
- Error type: `BuildWatchError` in `lib.rs`.
