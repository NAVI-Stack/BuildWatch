# BuildWatch — Agent Instructions

## What This Is

BuildWatch is a Rust-based universal file watcher and build daemon. It watches project files via Watchman and auto-rebuilds when changes are detected. Part of the NAVI Ecosystem but fully standalone.

## Architecture

Read `docs/architecture.md` before making changes. It is the canonical specification.

## Project Structure

```
buildwatch/
├── src/
│   ├── main.rs        # CLI entry (clap) — `buildwatch` binary
│   ├── bin/wr.rs      # Freshness gate — `wr` binary
│   ├── lib.rs         # Shared library root
│   ├── config.rs      # Config parsing (buildwatch.config.json)
│   ├── daemon.rs      # Background daemon lifecycle
│   ├── watcher.rs     # Watchman integration
│   ├── queue.rs       # Build queue with priority scoring
│   ├── builder.rs     # Build execution engine
│   ├── detector.rs    # Project type auto-detection
│   ├── state.rs       # Shared state (JSON files on disk)
│   ├── notifier.rs    # System notifications
│   └── output.rs      # Terminal formatting & agent detection
├── tests/             # Integration tests
├── docs/
│   └── architecture.md  # Canonical spec
├── Cargo.toml
└── buildwatch.config.json   # (generated by `buildwatch init`, not committed)
```

## Key Dependencies

- `watchman_client` — Facebook's official Rust Watchman client (async/tokio)
- `clap` — CLI framework
- `tokio` — Async runtime
- `serde`/`serde_json` — Serialization

## Build & Test

```bash
cargo build            # Debug build
cargo build --release  # Release build
cargo test             # Run all tests
cargo clippy           # Lint
cargo fmt              # Format
```

## Conventions

- Config file: `buildwatch.config.json` (JSON, not TOML)
- State dir: `/tmp/buildwatch/<project-hash>/` (Unix) or `%TEMP%\buildwatch\<hash>\` (Windows)
- All state file writes must be atomic (write to tempfile, then rename)
- One daemon per project root, never shared
- `wr` binary reads state files only — never writes, never starts daemon
